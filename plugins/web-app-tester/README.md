title: Web App Tester
description: Automated web app behaviour verification triggered from a GitHub PR or Issue using Playwright MCP. Finds the testable URL, runs a structured test plan, and posts a step-by-step report as a GitHub comment.
---

The **Web App Tester** verifies web app behaviour for a GitHub PR or Issue by running browser-based tests via **Playwright MCP**.

It is **PR/Issue-driven**: point it at a PR or issue number and it finds the testable preview URL from comments, uses the existing test plan (or generates one automatically), executes each step in a fresh browser session, and posts a structured test execution report directly on the PR or issue.

Single command: **`/test-web-app`**

---

## How It Works

```mermaid
flowchart TD
    A[/test-web-app pr N or issue N] --> B[Fetch PR/Issue content, comments, commits, linked issues]
    B --> C{Test URL found in comments?}
    C -- No --> D[Post 'no URL found' comment and STOP]
    C -- Yes --> E{Test plan found?}
    E -- Yes --> F[Use existing test plan]
    E -- No --> G[Auto-generate test plan and post comment]
    F --> H[Open fresh Playwright browser session]
    G --> H
    H --> I[Navigate to test URL]
    I --> J[Execute steps in order, snapshot after each]
    J --> K{Step passed?}
    K -- No --> L[Retry up to 3 times with 5s wait]
    L --> M[Mark PASSED / FAILED / BLOCKED]
    K -- Yes --> M
    M --> N[Close browser session]
    N --> O[Post test execution report as GitHub comment]
```

1. **Gather** — fetches full PR/issue body, all comments, commit messages, and linked issue content.
2. **Find URL** — scans content for a testable URL with labels like `Preview URL:`, `Staging URL:`, `Deploy preview:`. Posts a comment and stops if none is found.
3. **Find or generate test plan** — uses an existing structured test plan from comments (including ones generated by `test-strategist`), or auto-generates one from the PR/issue context.
4. **Execute** — opens a fresh Playwright browser session, navigates to the URL, and runs each step in order with snapshots.
5. **Report** — posts a single structured comment on the PR/issue with a step-by-step results table, failure details, and screenshots for FAILED/BLOCKED steps.

## Inputs

| Input | Source | Required | Description |
|---|---|---|---|
| `target_id` | Command argument | Yes | GitHub PR number (`pr 42`) or Issue number (`issue 88`) |
| Test URL | PR/issue comments | Yes | URL labelled `Preview URL:`, `Staging URL:`, `Deploy preview:`, etc. |
| Test plan | PR/issue comments | No | Structured numbered/bulleted steps; auto-generated if absent |

## Step Statuses

| Status | Meaning |
|---|---|
| ✅ PASSED | Step executed, expected outcome observed |
| ❌ FAILED | Step executed, expected outcome NOT observed |
| 🔴 BLOCKED | Step could not execute after 3 retries |

Overall result is **PASSED** when all steps pass, **FAILED** when steps fail but all were attempted, and **BLOCKED** when one or more steps could not execute.

---

## Auto-Generated Test Plan

If no test plan is found in the PR/issue, the agent generates one covering:
- The primary user-facing change
- Implied UI interactions (forms, navigation, state changes, error states)
- At least one happy-path and one edge/negative scenario

The auto-generated plan is posted as a comment before execution begins.

---

## Safety Rules

- **No URL found** → posts a comment and stops immediately
- **Production URL** (no `staging`, `preview`, `dev`, `test`, `localhost` in domain) → read-only mode; form submissions and destructive actions are skipped
- **Credentials** — never included in any posted comment
- **Browser session** — always closed after the run, regardless of outcome
- **Destructive actions** — never performed unless explicitly present in the test plan AND the URL is non-production

## Report Format

The plugin posts a single GitHub comment with this structure:

```
🤖 web-app-tester — Test Execution Report
URL tested: https://preview.example.com
Total: 5 | ✅ Passed: 4 | ❌ Failed: 1 | 🔴 Blocked: 0
Overall: FAILED

| # | Step | Status |
|---|------|--------|
| 1 | Navigate to /login | ✅ PASSED |
| 2 | Fill email and password, click Sign In | ✅ PASSED |
| 3 | Verify redirect to /dashboard | ✅ PASSED |
| 4 | Click New Order button | ❌ FAILED |
| 5 | Verify order form appears | ✅ PASSED |

---

### Failed / Blocked Steps

**Step 4 — Click New Order button**
Reason: Button not found after 3 retries — element `.btn-new-order` absent from DOM
Screenshot: captured at point of failure
```

## Prerequisites

- **Playwright MCP server** configured in `claude_desktop_config.json` (see `docs/mcp-config.md`)

## Quick Start

```bash
# Point Claude Code at the plugin
claude --plugin-dir /path/to/xianix-plugins-official/plugins/web-app-tester

# Then in the chat, trigger a test run
/test-web-app pr 42
```

Make sure your PR has a comment with a testable URL, for example:
```
Preview URL: https://pr-42--my-app.netlify.app
```
